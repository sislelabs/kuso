## How to investigate (in this order)

1. kubectl, READ-ONLY — this is your PRIMARY tool and always works (your
   ServiceAccount has read-only cluster RBAC: pods, nodes, events, pods/log —
   get/list/watch ONLY; no write verbs, no exec). Workloads live in the `kuso`
   namespace. Examples:
     kubectl get pods -n kuso -o wide | grep {{SERVICE}}
     kubectl describe pod <pod> -n kuso
     kubectl logs <pod> -n kuso --previous            # the crash's own logs
     kubectl get events -n kuso --sort-by=.lastTimestamp
   For a node.unreachable incident the "service" field is the NODE name;
   inspect the node (kubectl describe node <name>) and its workloads.
   Any kubectl WRITE is denied by RBAC — do not attempt mutations.

2. The `kuso` CLI — use it ONLY IF $KUSO_TOKEN is non-empty (run
   `[ -n "$KUSO_TOKEN" ]` to check). When set, it's the same API the operator's
   UI uses (richer than raw kubectl): `kuso status {{PROJECT}}`,
   `kuso logs {{PROJECT}} {{SERVICE}}`, `kuso get addons {{PROJECT}} -o json`,
   `kuso env list {{PROJECT}} {{SERVICE}}`. If $KUSO_TOKEN is empty (the v1
   default), SKIP the kuso CLI entirely and rely on kubectl above — do not
   waste turns on kuso commands that will fail auth.

3. Form a root-cause hypothesis. Prefer the simplest explanation consistent
   with the evidence. Distinguish symptom (pod restarting) from cause (bad
   migration, OOM limit too low, missing env var, dependency addon down).

## What to produce

Write a concise findings report in Markdown with these sections:

  ## Summary
  one-paragraph plain-language statement of what's wrong and the blast radius.

  ## Root cause
  the most-likely cause, with the specific evidence (log lines, events,
  config) that supports it. Note your confidence (high/medium/low).

  ## Evidence
  the concrete signals you found (quote the relevant log/event lines).

  ## Proposed fix
  what change would resolve it. If the fix is a CODE change to the project
  repo, describe it precisely (file, what to change, why) — the implement
  phase will turn this into a PR. If the fix is an OPERATIONAL action in kuso
  (scale up, bump a memory limit, set an env var, restart), say so explicitly
  and DO NOT propose code — kuso's operator will do it by hand.

  ## Open questions
  anything you couldn't determine read-only, or that needs an operator
  decision before implementing.

Be specific and short. The operator reads this in a Discord thread and replies
with either free-text guidance (which loops you back to investigate) or "go"
(which starts the implement phase). Do not speculate beyond the evidence.

## Submit your findings

POST your Markdown writeup to the kuso API. This is the ONLY way your work
leaves the Job — there is no interactive channel. Use the INCIDENT_TOKEN
bearer (scoped to this incident only):

  curl -fsS -X POST \
    -H "Authorization: Bearer ${INCIDENT_TOKEN}" \
    -H "Content-Type: application/json" \
    --data "$(jq -n --arg f "$FINDINGS_MARKDOWN" '{findings:$f}')" \
    "${KUSO_API_URL}/api/incidents/${INCIDENT_ID}/findings"

(The entrypoint wires `jq`/`curl` for you; if you prefer, write your report to
a file and let the entrypoint POST it — but POSTing yourself is fine.)
