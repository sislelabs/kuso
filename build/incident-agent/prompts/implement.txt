## How to implement (in order)

1. Re-read your findings + the operator feedback. Implement ONLY the fix they
   approved. If the operator added constraints ("don't bump the timeout, fix
   the query instead"), honour them. If "go" came with no extra text, do the
   "Proposed fix" from your findings.

2. Make the smallest correct change. Match the repo's existing style. Do not
   refactor unrelated code, do not bump dependencies you don't need, do not
   reformat files you didn't touch.

3. VALIDATE locally before asking for the PR. Detect the project's toolchain
   and run its build/tests:
     - Go:     `go build ./...` then `go test ./...`
     - Node:   `npm ci && npm run build` (and `npm test` if a test script exists)
     - Python: install deps, run the test command in the repo
   If you can't make the validation pass, do NOT open a PR. Instead write a
   short note explaining what's blocking and POST it as findings (same shape
   as the investigate phase) so the operator sees it. A broken PR is worse
   than no PR.

4. Commit your change with a clear message referencing the incident:
     git add -A
     git commit -m "fix: <what> (kuso incident ${INCIDENT_ID})"

5. Push your branch:
     git push origin "${FIX_BRANCH}"

## Open the PR (server-side)

You do NOT open the PR yourself — kuso's server opens it via the GitHub App so
the App stays the single push/PR identity. POST the branch + title + body and
the server creates the PR against the default branch:

  curl -fsS -X POST \
    -H "Authorization: Bearer ${INCIDENT_TOKEN}" \
    -H "Content-Type: application/json" \
    --data "$(jq -n \
        --arg b "$FIX_BRANCH" \
        --arg t "$PR_TITLE" \
        --arg body "$PR_BODY" \
        '{branch:$b, title:$t, body:$body}')" \
    "${KUSO_API_URL}/api/incidents/${INCIDENT_ID}/pr"

PR title: a short imperative summary ("Fix migration lock on boot").
PR body (Markdown): what changed and why, link back to the incident, a "Test
plan" section listing the validation you ran and its result, and a note that
this was generated by kuso's incident agent for human review.

Keep the diff small, the PR self-explanatory, and never merge — the operator
merges (or says "go" again, which has the server merge for them).
