# kuso web UX + product-gap review — 2026-05-13 (post v0.10.0)

Walked the app as three personas: solo founder on day 1, small-team admin
on day 30, technical operator at 2am during a bad deploy. Findings are
grounded in actual source under `web/src` — no fabrication. P0/P1/P2 +
suggested fix style (drop / build / honest copy) attached to each.

Scope reminder: kuso is single-tenant. Out-of-scope items (DB branching,
Grafana clone, edge functions, WAF, multi-region, native error tracker,
Vercel parity) are not flagged here.

## Top of the list (read these first)

### P0-A · Onboarding "Step 1 admin-only" leaves non-admin first-users stranded
File: `web/src/app/(app)/welcome/page.tsx:194-215`

Non-admins who land on /welcome get a `installURL.isError` branch that
prints a sentence ("Ask a team admin to install it, or skip and start
without a repo") plus a `Skip` button. There is no way to:
- contact the admin from inside kuso,
- request access,
- understand that being a non-admin is what's happening (the copy says
  "admin", but a fresh signup has no concept of "the admin").

Day-1 path for a fresh signup invited via group memberships only:
1. land on /welcome,
2. click Install on GitHub → 403,
3. read "ask a team admin," dead end,
4. click Skip → bounce through /projects → /welcome loop is only
   broken by `sessionStorage` (`projects/page.tsx:43-57`); without
   memberships the user lands on /awaiting-access with no breadcrumb
   home.

Fix sketch (build): when `installURL.isError && status===403`, show a
one-tap "Notify your admin" affordance that POSTs a server-side
notify-event (operators see it in the bell feed). Pair with a copy
change: replace "Install on GitHub" with "Connect a repo (admin needed)"
so the gate is legible before the click.

### P0-B · `/welcome` skip + "start without a repo" lands users at /projects/new with no usable next step
Files: `web/src/app/(app)/welcome/page.tsx:213, 254-261`, `web/src/app/(app)/projects/new/page.tsx:170-176`

Step 1's `Skip` button + Step 2's "Start without a repo" both route the
user into the project-create form. After the form submits, the user is
redirected to `/projects/<name>`, which renders the canvas empty-state
("Empty project" with `+ Add service`). `Add service` → the GitHub
repo-picker view, which immediately re-shows the same `Install on
GitHub` requirement they just skipped (`services/new/view.tsx:159-186`).
The "skip the GitHub step" path is a circle.

Fix sketch (build): on a fresh project with no GitHub install, the
canvas empty-state should surface "import a docker image instead" or
"connect git later" as a peer action to `Add service`. Today the only
two CTAs are `+ Add service` (needs GitHub) and `+ Add addon`, leaving
zero ways to spin up the first compute without a repo.

### P0-C · Build-failure path doesn't tell the user *why* the build failed before scrolling kaniko logs
Files: `web/src/components/service/overlay/ServiceDeploymentsPanel.tsx:260-345`, `web/src/components/logs/LogStream.tsx`

The Deployments tab shows a red FAILED badge, the commit SHA, the
duration, and "by you / via webhook". To see *what* broke, the user has
to expand the row, which mounts a `<LogStream env="build:<id>"/>` and
streams the full kaniko log (typically 200-600 lines, mostly noise from
the base image pull). There's no summary, no error highlight, no
"jump to first error". The deploy panel happily renders a wall of green
INFO lines from `kaniko --no-push` and buries the offending compile
error at line 412.

Operators using kuso during an incident at 2am should not be
re-implementing `grep -E '(error|fatal|panic)'` in their head. The
ErrorRow component on the Errors tab (`ServiceErrorsPanel.tsx`) already
proves we can extract structured causes from log lines — none of that
machinery is wired into the build view.

Fix sketch (build): server side, scan build logs for the first
error-class line (`error:|FATAL|panic:|undefined reference|EXIT 1|the
command '\S+' returned a non-zero code|ModuleNotFoundError`) and stash
it on the BuildSummary as `errorHint` + `errorLine`. Render it as a
sticky red banner above the build-logs viewport with "jump to line N"
that scrolls the `LogStream` to that line. Falls back to today's wall
when no hint matched.

## Solo founder, day 1

### P1-1 · Repo picker shows "<owner>/<repo>" with no signal of which org-install owns it on /welcome
File: `web/src/app/(app)/welcome/page.tsx:328-360`

Step 2 lists every repo from the *first* installation by default (`233`)
and only surfaces an org picker when `installations.length > 1`. Users
with one personal install + one org install will see exactly one org's
repos until they click the secondary chip. The active-install border
state is the only indicator, and `installations[0]?.id` is
deterministic-but-unstable across reloads.

Fix sketch (build): always render the org-picker row even with one
install, but as a single static label ("from `<owner>`") with a
"see other installs" link. The `installID` lookup at the AddServiceView
(`services/new/view.tsx:54-58`) already does this correctly — flatten
across all installs into one filterable list. Mirror that pattern on
/welcome.

### P1-2 · "URL preview" on /projects/new uses literal `web` as the example slug — misleading for monorepos and single-service projects
File: `web/src/app/(app)/projects/new/page.tsx:60-156`

The preview pill renders `https://web.{baseDomain}` whether or not the
user is planning to call their service "web". A user creating a project
for a single API service will think "web" is a kuso reserved word.

Fix (honest copy): change `exampleService` to use the project name as
the slug, e.g. `<your-service>.my-product.example.com` with literal
italic placeholder text. Once a service is added the canvas shows the
real domain — the preview only needs to teach the shape.

### P1-3 · /welcome Step 3 over-promises "60–90s first build"
File: `web/src/app/(app)/welcome/page.tsx:426-431`

> The first build typically finishes in 60–90s.

First-time kaniko builds on a fresh kuso install pull the entire base
image (700MB-2GB), set up the kaniko cache, and resolve a fresh
`npm install` / `pip install` / `bundle install`. On a cold cluster
this routinely runs 3-6 minutes. The "60-90s" claim invites users to
think kuso is broken at the 100s mark and reload — losing buffered
build logs (`LogStream` re-establishes the stream on remount).

Fix (honest copy): "The first build typically finishes in 2-5 minutes
(subsequent pushes are faster — kaniko caches layers)." Same affordance,
honest expectation.

### P1-4 · Service-create wizard hides runtime-detection failure
File: `web/src/app/(app)/projects/[project]/services/new/view.tsx:75-91`

`detect.mutateAsync().catch(() => { /* leave defaults */ })` silently
swallows detection errors — including 404 on a private repo where the
installation doesn't actually have read access. The user sees "Service"
section fill in with `runtime=dockerfile, port=""` and no signal that
the autodetect actually ran. They press Add, hit a build failure 4
minutes later, and only then learn kuso never saw their repo.

Fix sketch (build): catch the error and render an inline orange chip
above the runtime picker — "Couldn't inspect repo: <err>. Continuing
with defaults". Reuse `DetectedEnvBanner`'s amber treatment for
visual continuity.

## Small-team admin, day 30

### P1-5 · Project Settings page is the lone hold-out without a unified SaveBar — silent edit loss on tab/back-button
File: `web/src/app/(app)/projects/[project]/settings/view.tsx:91-241`

ServiceOverlay (`ServiceOverlay.tsx:498-559`) and AddonOverlay
(`AddonOverlay.tsx:268-318`) both render a unified floating SaveBar
driven by `useOverlayDirty`. Project settings still uses a single
bottom-right `Save changes` Button and no dirty-state pip. A user
who edits `description`, `previewsEnabled`, and `alwaysOn` then clicks
the breadcrumb back to canvas loses every change with zero indication.

Fix sketch (build): port the `useOverlayDirty` pattern to a
page-scoped equivalent. The project-settings page is a route, not an
overlay, but the same dirty-map + floating bar idea applies.
Visually consistent across the three places that hold form state.

### P1-6 · AddonOverlay SettingsTab renders inline per-section footer Save *and* the unified SaveBar at the same time
File: `web/src/components/addon/overlay/SettingsTab.tsx:274-294, 563-580`

The Configuration section + Placement section each ship their own
`<footer>` with Save + Reset buttons (lines 274-294, 563-580), *and*
the parent overlay renders the unified SaveBar from the
`useAddonOverlayDirty` registration. On a 1280×800 viewport with
dirty state, the user sees Save in three places: inline configuration
footer, inline placement footer, floating overlay SaveBar. The
EnvVarsEditor fix (U-P0-D) deliberately removed exactly this duplication
for the service variables panel — the addon Settings tab missed the
same migration.

Fix sketch (drop): delete the inline `<footer>` blocks in
ConfigurationSection and PlacementSection. The unified SaveBar
already handles the operation correctly.

### P1-7 · "Rollback" requires hunting through Deployments for a build with SUPERSEDED badge
File: `web/src/components/service/overlay/ServiceDeploymentsPanel.tsx:315-317, 418-490`

The rollback affordance is conditional on `s === "superseded"` —
rendered as a tiny "rollback" pill inline on the row. Users looking
to roll back from a bad deploy:
1. Open service overlay → Deployments tab,
2. Spot the most-recent green build above current red one,
3. Hover row, find the muted `rollback` chip on the right.

For a service with 50 builds in history and pagination, "rollback to
last good build" is a multi-click hunt. Most CD tools expose this as
a single shortcut.

Fix sketch (build): add a "Rollback to last green" button next to
Redeploy in the deployments header (`ServiceDeploymentsPanel.tsx:185-189`),
enabled when the latest non-active build is a superseded build. Behind
the same `useCan(Perms.ServicesWrite)` gate.

### P1-8 · No env-var search / filter inside EnvVarsEditor
File: `web/src/components/service/EnvVarsEditor.tsx:486-642`

A service with 40 env vars (typical for a Rails or Django app) renders
40 rows with no filter input. The bulk-mode textarea is the workaround
("paste then ctrl-F") but it's lossy on secret-backed entries (line
197-209: comment-out as `# NAME=<from secret>`). For everyday
"where is DATABASE_URL set" the cmd-K env-var index
(`CommandPalette.tsx:177-194`) wins — it surfaces "DATABASE_URL →
web" cross-service. But inside one service's editor, there's no inline
filter.

Fix sketch (build): inline `<input placeholder="filter vars">` above
the rows grid, substring-matches name+value. Cheap; mirrors the
LogStream filter pattern.

### P1-9 · Cross-service "duplicate service to another project" doesn't exist
Files: `web/src/components/service/overlay/settings/*`, `web/src/components/canvas/ProjectCanvas.tsx`

Day-30 small-team scenario: "stage" project's `api` service has a
20-key env-var matrix + custom domain + placement labels + custom
runtime command. Now needed in a separate `staging-eu` project. Today
this is 12+ clicks of manual reconstruction — there's no clone/copy
action anywhere.

Fix sketch (build): add `Settings → Danger zone (rename to "Clone &
delete")` action "Duplicate this service to…" that takes a target
project + service name. Server-side copies spec + envVars (secret-ref
rewrites point at the target project's addons, fall back to literal
nulls with a warning chip on each unresolved ref).

### P1-10 · No "compare two builds' env diffs" — needed when a deploy regression is suspected
File: `web/src/components/service/overlay/ServiceDeploymentsPanel.tsx`

The deployments list shows builds without a way to compare env-var
state across two builds. Common day-30 question: "what env vars
changed between the last green deploy and the broken one?" today
needs git-log spelunking or memory.

Fix sketch (build): on the build row, add a "Compare to active"
context-menu action that opens DiffConfirmDialog-style modal showing
env-var diff between that build's snapshot and current. Requires the
server to stamp the env snapshot on each KusoBuild — modest extension
to the existing CR.

## Technical operator, bad deploy at 2am

### P1-11 · ServiceOverlay's "Errors" tab pings since=24h by default — wrong window for the operator who just got paged on a 60s-old incident
File: `web/src/components/service/overlay/ServiceErrorsPanel.tsx:34, 16-21`

When the user opens Errors during an incident, the default `since=24h`
hides nothing in the typical pre-incident window. They see 24 hours of
noise, scroll-hunt for the one that started 5 minutes ago, then change
to "1 hour" anyway.

Fix sketch (honest copy + build): default since to "1h" for fresh
opens. Persist the user's last-chosen value in sessionStorage like
ServiceOverlay does with its tab selection.

### P1-12 · Build log streamer says "waiting for logs…" forever when build pod was never scheduled
File: `web/src/components/logs/LogStream.tsx:201-209`, `web/src/components/service/overlay/ServiceDeploymentsPanel.tsx:360-371`

For a build stuck in `pending` (no nodes match build pool placement,
or kaniko-build-secret missing, etc.) the LogStream prints "waiting
for logs…" indefinitely. There's no hint of "the pod was never
scheduled, here's the kube event explaining why."

Fix sketch (build): when a `useBuild()` returns `phase=pending` AND
the LogStream reports zero lines after 30s, surface the pod's
`Events` (Failed scheduling / no nodes available / image pull error)
above the empty log viewport. The server already has access to these
via `kubectl get events --field-selector involvedObject.name=<pod>`;
expose at `/api/projects/<p>/services/<s>/builds/<id>/events`.

### P1-13 · No "tail live logs across all envs of this service" view
File: `web/src/components/service/overlay/ServiceLogsPanel.tsx:122-134`

The env dropdown is single-select. During an incident the operator
often wants to grep for an error across `production` + `preview-pr-N`
to determine if the bug is a regression. Today they have to flip the
dropdown, copy results, flip again.

Fix sketch (build): add an `(all envs)` value to the env dropdown
that omits the `env` query param entirely. Server already supports
this — the FTS5 query is filtered when `env` is set and unfiltered
when blank.

### P1-14 · Service URL pill on canvas obscures HTTP-vs-HTTPS scheme on cluster-default domains
File: `web/src/components/canvas/ServiceNode.tsx:410-454`

`UrlPill` does `url.replace(/^https?:\/\//, "")` to fit the pill,
which removes the user's only visual signal that the cert path is
HTTPS. A service that legitimately serves HTTP (internal=true is the
"internal only" path; but a custom domain with `tls: false` flag from
`spec.domains[0].tls` also renders HTTP) reads identically to its
HTTPS sibling. During an "is my cert expired" incident, this is the
exact signal you want at-a-glance.

Fix sketch (honest copy): prepend a tiny `🔒` or `⚠ http` glyph
inside the pill when `customTLS === false` so the scheme is visible
without re-expanding the URL. Or render `http://` literally when the
custom-domain's TLS is off, since that's a deliberate exceptional
config.

## Information density, cross-cutting

### P1-15 · `LogStream` filter input is 1 line — accepts substring OR regex via `/prefix` but never tells you which mode you're in
File: `web/src/components/logs/LogStream.tsx:25-38, 154-167`

The matcher infers regex mode from a leading `/`; bad regex falls
through to a no-op filter silently (line 32). A user typing `/[unclosed`
sees their filter "match nothing" with no explanation. Returns `null`
on invalid regex but renders the filter UI as if it's filtering normally.

Fix sketch (build): when the matcher is null because of regex parse
failure, render a small "invalid regex — showing all lines" chip next
to the n/total counter. Cheap; uses the same matched-count slot.

### P2-16 · "Ready", "Active", "Live", "Up" are used interchangeably across the UI
Examples:
- `ServiceOverlay.tsx:572` — `Active` label on the StatusDot
- `ServiceNode.tsx:142` — `"active"` status value renders an emerald border
- `projects/page.tsx:284-296` — "X of Y services live" / "healthy / degraded / down"
- `ServiceDeploymentsPanel.tsx:119` — `ACTIVE` badge for the running build
- `MobileIncidentView.tsx:67-69` — uppercase `env.status.phase` (which can read `READY`, `RUNNING`, `ACTIVE`, …)

Five sources of truth, five vocabularies. The data model has clear
states (env.ready, build.status, replicas.ready) but the rendered
labels float — a user can't reason "is this thing live" by matching
strings.

Fix sketch (honest copy): pick one vocabulary per scope and document
it in `docs/REDESIGN.md` or similar. Suggestion: for env runtime use
`Active / Building / Failed / Sleeping / Idle`; for builds use
`Active / Superseded / Failed / Building / Cancelled`; for cards use
"X of Y live" only when X is `ready replicas > 0`. The current code
mostly agrees with this — it just renders inconsistent strings to
the user.

### P2-17 · Bell-icon feed swallows error-class events into "no events yet"
File: `web/src/components/layout/TopNav.tsx:558-576`

`feed.isError` falls through to `feed.error.message` — but `enabled:
false` + on-demand `void feed.refetch()` (line 503) means the popover
opens before the request completes; the first render shows
`feed.isPending` → "Loading…" but the empty-state copy
("No events yet…") races. On a real 500-from-server the user sees
either the loading state or "no events yet" depending on the
race — never the actual error. Hard to debug.

Fix sketch (build): keep `enabled: false` for cost but call refetch
synchronously inside `onOpenChange` and treat `feed.fetchStatus ===
"fetching"` as the loading state, not `isPending`. Surface
`feed.isError` distinctly with a Retry button.

### P2-18 · Canvas keyboard shortcuts (j/k/r/l/v/s) work, are great, and are documented only in a single `?` toast
File: `web/src/components/canvas/ProjectCanvas.tsx:370-452`

The shortcuts are real power-tools (focus / open / logs / vars /
settings / redeploy by row). Discovery is one `?` keypress, no
visible affordance anywhere else. CommandPalette doesn't list them.
The canvas itself doesn't render a small "press ?" hint.

Fix sketch (build): tiny `kbd>?<` hint in the EdgeControlsPanel's
bottom-right cluster (or near the canvas Controls). Already-existing
infrastructure — one more pill on a row that already has chips.

### P2-19 · `kuso-service-overlay-tab` persists across overlay opens — fine for power users, surprising for first-timers
File: `web/src/components/service/ServiceOverlay.tsx:169-190`

Opening a service overlay for the *first time* lands on Deployments.
Then opening it again lands on whatever tab was last left. A user who
clicks Logs once for service A, then clicks service B on the canvas,
lands on Logs for B with no preserved state — fine. But the
expectation that "click service → see deploys" is broken once you've
ever opened Logs.

Fix sketch (drop): persist tab choice only inside the same overlay
session (clear `kuso-service-overlay-tab` on close) so each fresh
service open lands on Deployments unless a `?tab=` deeplink overrides.
The current behaviour is "remember tab forever in sessionStorage" —
midground that surprises both first-timers and power users.

### P2-20 · DetectedEnvBanner suggests env-var names from log-scan but doesn't link to the line in the log
File: `web/src/components/service/EnvVarsEditor.tsx:933-1028`

The crash-hint variant ("Recent crash mentions an env var that
isn't set") shows the `lastLine` truncated to the row width. There's
no "see in log" link, so the user has to manually flip to the Logs
tab + search for the same string. Two clicks that should be one.

Fix sketch (build): make each hint row a button-link to
`/projects/<p>?service=<s>&tab=logs&q=<lastLine>` (or pass the var
name as the FTS5 query) so the user lands on the log line that
triggered the hint. The `?tab=` deeplink machinery already exists.

### P2-21 · Mobile incident mode covers services only; addon outages on phones have no surface
Files: `web/src/components/project/MobileIncidentView.tsx`, `view.tsx:174-182`

`MobileIncidentView` only iterates services. If the on-call gets paged
for "postgres pod CrashLooping" they open kuso on their phone, see
service cards, no addon card, no surface to read postgres logs / open
SQL / kick a restart. AddonOverlay isn't reachable from any phone-
visible affordance.

Fix sketch (build): extend MobileIncidentView to render compact
addon cards too. Each card has "Open inspector" → AddonOverlay (which
should already adapt full-screen on phones — verify), "Restart" if
operator has the perm, "Open logs" (already works for the StatefulSet
pod under the hood).

## Smaller findings (P2)

### P2-22 · `ServiceLogsPanel`'s default 200-line limit silently truncates without telling the user
File: `web/src/components/service/overlay/ServiceLogsPanel.tsx:67-77`

Pulls 200 lines; with `committed.q === ""` polls every 10s. No
visible "showing 200 of N" indicator on the result page. A user
investigating a high-volume service can mistake 200 for "all of them"
and conclude the issue isn't logged.

Fix sketch (honest copy): if `lines.length === 200` (the limit),
render "showing 200 most recent — paginate / refine your query for
older" above the table.

### P2-23 · `DriftBanner` "Saved Ns ago — pod started Ms after save" disappears after 60s
File: `web/src/components/service/EnvVarsEditor.tsx:1209-1224`

The banner shows for two minutes (line 1211: `sinceSave < 120`) which
is great UX during the rollout. After 2 minutes it just vanishes — no
"the change is live" persistent indicator. A user who walks away from
the screen for 5 minutes comes back to a clean editor and can't tell
if their save actually rolled.

Fix sketch (build): keep a quiet (low-emphasis) "last rolled out X
ago" line at the top of the editor permanently, sourced from
`drift.lastRolloutAt`. The dramatic 2-minute banner is the
celebratory state; a tiny line below is the steady-state truth.

### P2-24 · `CommandPalette` redeploys silently — no destructive confirm and no diff
File: `web/src/components/command/CommandPalette.tsx:105-113, 144-171`

`Redeploy <name>` fires `triggerBuild` without confirmation. cmd-K is
the keyboard-first power-user surface, so "no confirm" is defensible,
but a missed enter on a typo (typing `r` to fuzzy-match
"role-management" lands on `Redeploy api`) triggers a production
build. Same `Tail logs` next to it is a no-op safe action — they
visually read the same.

Fix sketch (honest copy): label as "Trigger deploy · <name>" so the
verb makes the side-effect explicit, and consider adding a tiny
" ⏎ to confirm" hint inline.

## Off-list product-gap calls (day 30 essentials)

The brief asked for "what's missing solo founders actually need on day
30". Beyond the P0/P1 items above:

- **Env-group templates** (preview/staging/prod share config matrix
  with selective overrides): no surface today. `NonProdBanner`
  (`view.tsx:240-282`) describes the "cloned from production" semantic
  but the matrix is invisible — there's no tabular "envs × env-vars"
  view to spot drift across staging vs production.

- **"Restore backup to a NEW addon"**: `BackupsTab.tsx:140-180` allows
  cross-addon restore into a *sibling that already exists*. Common
  recovery pattern is "scratch the bad state on prod-pg, restore
  yesterday's dump into a NEW addon, point app at it via env-var
  swap." Today requires: 1) Add addon, 2) Backups tab, 3) restore into
  new, 4) Variables tab, 5) swap ref. Four overlays. Should be one
  flow: "Restore as new addon" button on the backup row.

- **Cross-project secret store** doesn't exist. Project-shared
  secrets (`/projects/<p>/settings`) and instance-shared secrets
  (`/settings/instance-secrets`) cover within-project and global —
  there's no "this org has 3 projects sharing the same SENTRY_DSN"
  scope between them. Pattern bites teams the moment they split
  monorepo into multi-project setup.

- **"Recent activity for this service"** lives nowhere. Activity feed
  is project-wide (`ActivityFeed.tsx`); the service overlay has
  no per-service timeline. Common ops question: "what changed on
  this service in the last 24h?" today requires the audit log +
  client-side filter.

## Summary

Of 24 findings: 3 × P0 (onboarding circle, no admin-notify path,
build-failure-cause invisible), 11 × P1 (admin daily flow + incident
flow gaps), 10 × P2 (polish, copy, micro-discoverability).

The platform's load-bearing patterns (`useOverlayDirty` SaveBar,
`?tab=` deeplinks, drift-banner state machine, cmd-K env index) are
strong and well-considered. The gaps that remain mostly fall into
three buckets:

1. **First-deploy success rate** is throttled by the GitHub-App-admin
   gate (P0-A) and the runtime-detect silent-fallback (P1-4).
2. **Build-failure observability** is the single biggest 2am pain
   point (P0-C, P1-12, P2-22) — no error summary, no first-error
   jump, no pod-scheduling hints.
3. **Save-bar consistency** is 80% there. Project settings (P1-5)
   and addon Settings tab inline footers (P1-6) are the two
   stragglers; landing both finishes the unified pattern.
