# Kuso Public-Egress Fix — Design

**Date:** 2026-05-20
**Status:** Approved
**Repo:** kuso (`/Users/sisle/code/work/kubero-setup/kuso`)

## Problem

kuso's `kusoproject` Helm chart renders a per-project `default-deny`
NetworkPolicy plus companion allow-policies. One of them,
`<project>-allow-public-egress`, permits egress to the public internet
(`0.0.0.0/0` minus RFC-1918 + cloud-metadata) — but **only for pods carrying
the label `kuso.sislelabs.com/network-egress-public=true`**.

The `kusoenvironment` chart, which renders runtime *service* pods, has no way
to set that label. Only *build* pods get it (`buildcontroller.go:442`). As a
result, every kuso service that calls an external API (Discord, Stripe, an
LLM provider, an outbound webhook) is silently blocked with `ECONNREFUSED`
once per-project NetworkPolicies exist.

This was discovered when the `distill` project's Discord OAuth login and
Discord bot both failed with `connection refused` to `discord.com`. A manual
`kubectl patch` of `distill-allow-public-egress` (dropping the label
requirement from its `podSelector`) restored egress as a stopgap; that patch
is reverted whenever the operator next reconciles the project, so a permanent
fix in kuso is required.

## Goal

Give runtime service pods a supported, chart-driven way to receive the
`network-egress-public=true` label, so public internet egress works by
default and every existing service is healed on upgrade.

## Decisions

- **Default:** public egress is **ON** by default. Apps normally call
  external APIs; the silent default-deny was the bug. Services opt *out*.
- **Scope:** a **per-service** field on `KusoService`, propagated to each
  `KusoEnvironment` — mirrors the existing `Internal bool` field exactly.
- **Field name:** `PrivateEgress bool` (`json:"privateEgress,omitempty"`).
  Because the default is ON, the field is an opt-*out*; a plain `bool` whose
  Go zero value (`false`) means "egress allowed" needs no migration for
  existing CRs.
- **CLI:** `kuso project service add` gains a `--private-egress` flag (v1).

## Changes

### 1. `KusoServiceSpec` — `server-go/internal/kube/types.go`

Add `PrivateEgress bool` (`json:"privateEgress,omitempty"`) with a doc
comment explaining: when true, the service's pods are namespace-internal
only (no public internet egress); default false = egress allowed. Place it
near `Internal` for discoverability.

### 2. `KusoEnvironmentSpec` — `server-go/internal/kube/types.go`

Add the mirrored `PrivateEgress bool` (`json:"privateEgress,omitempty"`),
documented as server-managed (propagated from the service spec), exactly
like the existing `Internal` mirror field.

### 3. Propagation — `server-go/internal/projects/propagate.go`

`propagateChangedToEnvs` is the single chokepoint that mirrors service-spec
edits onto env CRs. Add `privateEgress` to the `changedFields` set and to the
propagation logic so that when a service's `PrivateEgress` changes via
`PatchService`, every env CR for that service is updated. Mirror the existing
`internal` field handling case-for-case.

### 4. `kusoenvironment` Helm chart

- `values.yaml`: add `privateEgress: false` with a comment.
- `templates/deployment.yaml`: in the **pod template's** `metadata.labels`
  block (the `spec.template.metadata.labels`, ~line 39-41 — NOT the
  Deployment's own `metadata.labels`), conditionally add
  `kuso.sislelabs.com/network-egress-public: "true"` **unless**
  `.Values.privateEgress` is true:
  ```yaml
  {{- if not .Values.privateEgress }}
  kuso.sislelabs.com/network-egress-public: "true"
  {{- end }}
  ```
  This label is the single switch the `kusoproject` chart's
  `allow-public-egress` policy keys on.

The server must also write `privateEgress` into the `kusoenvironment` Helm
values when rendering/applying the release (wherever env-CR fields are
currently mapped to chart values — same code path as `internal`).

### 5. CRD schemas

`KusoService` and `KusoEnvironment` are real CRDs. Their schemas live in:
- `operator/config/crd/bases/application.kuso.sislelabs.com_kusoservices.yaml`
- `operator/config/crd/bases/application.kuso.sislelabs.com_kusoenvironments.yaml`
- the bundled `dist/crds.yaml`

If the CRDs are generated by `controller-gen` from Go markers, updating the
Go structs (changes 1-2) and regenerating is the correct path; the plan must
run the generation target and verify `dist/crds.yaml` is regenerated/synced
to match `operator/config/crd/bases/`. If any schema is hand-maintained, the
`privateEgress` boolean property must be added by hand to both
`spec.properties` blocks.

### 6. CLI flag — `cli/`

`kuso project service add` gains `--private-egress` (bool, default false).
When set, the created `KusoService` carries `privateEgress: true`. Follow the
existing flag-wiring pattern used by sibling flags such as `--internal` /
`--runtime` / `--port` on the same command.

## What does NOT change

- The `kusoproject` NetworkPolicy chart — `allow-public-egress` already keys
  off the `network-egress-public=true` label. We are only finally giving
  runtime pods a way to carry it.
- The egress block-list CIDRs (RFC-1918 + cloud-metadata) — unchanged.
- No per-service *destination* allow-listing; egress remains all-or-nothing
  public, as today.

## Testing

- **Chart golden tests** (`render_test.go` / `golden_test.go`): add cases
  asserting the `kuso.sislelabs.com/network-egress-public=true` pod label is
  present by default and absent when `privateEgress: true`.
- **Propagation unit test**: mirror the existing `internal`-field propagation
  test — a `PatchService` toggling `PrivateEgress` updates all env CRs.
- **CLI**: verify `--private-egress` sets the field on the created service.

## Rollout

1. Implement changes 1-6, all tests green.
2. Build + release kuso; deploy the new operator + server to the cluster.
3. The `distill` project reconciles: the `kusoenvironment` chart now stamps
   `network-egress-public=true` on distill's runtime pods, so
   `allow-public-egress` covers them through the unmodified policy selector.
4. Revert the manual `kubectl patch` on `distill-allow-public-egress` so the
   live policy matches a clean chart render.
5. Verify distill: Discord OAuth login succeeds, the bot connects to the
   Discord gateway, `/api/proxy/*` calls work.

## Risks & Mitigations

- **Existing services on upgrade:** every existing `KusoService` CR has
  `PrivateEgress=false` (Go zero value) → all runtime pods receive the label
  → all regain egress. This is the intended healing outcome; no per-service
  action required.
- **CRD drift:** `dist/crds.yaml` must stay in sync with
  `operator/config/crd/bases/`. The plan must verify both after regeneration.
- **Pod-vs-Deployment label placement:** the label MUST go on the pod
  template's `metadata.labels`, not the Deployment's own metadata —
  NetworkPolicy `podSelector` matches pod labels. Getting this wrong renders
  the fix inert.

## Success Criteria

- A newly-created service reaches the public internet with no extra config.
- A service created with `--private-egress` (or `privateEgress: true`) does
  NOT carry the egress label and is namespace-internal only.
- All existing services regain egress after the operator upgrade.
- Chart golden tests and propagation tests pass.
- `dist/crds.yaml` and `operator/config/crd/bases/` agree.
- distill's Discord login and bot work after rollout, with the manual patch
  reverted.
