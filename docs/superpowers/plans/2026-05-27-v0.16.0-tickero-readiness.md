# v0.16.0 — Tickero migration readiness

Driver: migrate `bileti-official` to kuso. Both kuso and Tickero are pre-live, so we can land schema changes without back-compat shims. PlanetScale Postgres is the prod DB; kuso-managed Postgres is staging only.

## Scope (4 items)

| # | Item | Why for Tickero | Effort |
|---|------|-----------------|--------|
| 1 | Release hooks — `KusoService.spec.release.command` | replaces the `initContainer: migrate up` pattern; no replica race, no probe thrash | 2-3 days |
| 2 | External-DB scheduled backup | snapshot PlanetScale → cluster S3 so a managed-provider outage doesn't lock us out of the data | 0.5 day |
| 3 | Cron onFailure watcher | refund-deadline + voucher-expiry crons silently failing = revenue leak; schema shipped in v0.15.0 | 1 day |
| 4 | Per-path `wakeOn` exclusion | keep `/api/v1/payments/notify` warm even when sleep is on globally; ePay retry-timeout safety | 0.5 day |

Total: ~5 focused days. Single ship as v0.16.0.

## Item 1 — Release hooks

### Schema

`KusoService.spec.release`:

```yaml
spec:
  release:
    command: ["./bin/migrate"]      # required when release block present
    timeoutSeconds: 900              # default
    envFromSecrets: []               # default: same as KusoEnvironment.envFromSecrets
```

Mirrored to `KusoEnvironment.spec.release` via the propagation step.

Image: always the **new build's image** — release hooks for Tickero are `migrate up` and that has to run against the binary about to ship. No image override field in v1. Add later if needed.

### Splice point

`server-go/internal/builds/builds.go:promoteImage` (around line 2060). Today:

```
build succeeds → patch env.spec.image.tag → operator rolls deployment
```

Becomes:

```
build succeeds
  if effective service.spec.release.command is set:
    create Job <env-name>-release-<short-tag>:
      image = b.Spec.Image.Repository:b.Spec.Image.Tag
      env = env.Spec.envVars (the resolved set)
      envFrom = env.Spec.envFromSecrets
      command = release.command
      activeDeadlineSeconds = release.timeoutSeconds
      backoffLimit = 0  (one shot — fail fast, don't retry migrations)
    poll Job up to release.timeoutSeconds:
      - succeeded → patch env.spec.image.tag (today's behavior)
      - failed/timeout → mark build phase=release-failed,
                          emit notify event, do NOT patch tag
  else fall through (today's behavior)
```

### Job idempotency

Name = `<env>-release-<short-tag>`. Re-deploys of same tag → Job exists, succeeded, skip. Re-builds of same SHA produce same short-tag → still works.

### New package `server-go/internal/releaserun/`

- `Run(ctx, env, image, opts) (Result, error)` — creates Job, polls, returns terminal state.
- `Logs(ctx, env, run) (io.ReadCloser, error)` — tails the Job's pod logs.
- Owner: the `KusoEnvironment` (so Job is GC'd if env is deleted).

### Failure semantics

- Job non-zero exit → `phase=release-failed` on the build CR, notify event `build.release_failed` with title `migrate up failed for <service>`.
- The env's existing pods keep running on the previous image. **Nothing rolls.**
- Re-trigger: user fixes migration, pushes a new commit, new build → new release Job → new attempt. No special "skip release" escape hatch in v1.

### CLI

- `kuso release-log <project> <service> [--env production]` — tails the most recent release Job for the env.
- `kuso build trigger <project> <service> --skip-release` — admin escape hatch.

### Web

- Service overlay → Settings tab → new "Release" section: text input for command (multi-arg, one per line or comma-split), timeout slider, "Skip release on next deploy" toggle.
- Build card on the deployments tab: when a release Job ran, show "Migration: ✅ 32s" / "❌ Migration failed — view log" with deep-link to logs.

### EDIT_SAFETY row

```
| `release` | ✅ Yes, on next deploy | Takes effect at next build. Existing pods are unaffected. Removing the field disables the release step on next deploy. |
```

### Smoke test (papelito on the test cluster)

2. Trigger a build. Confirm Job runs, succeeds, image promotes.

## Item 2 — External-DB scheduled backup (PlanetScale-aware)

### Schema

Reuse existing `KusoAddon.spec.backup.{schedule, retentionDays}`. The new behaviour kicks in when `spec.external` is also set.

### Chart

Extend `operator/helm-charts/kusoaddon/templates/backup-cronjob.yaml` with a fourth branch:

```yaml
{{- if and .Values.backup.schedule .Values.external (eq .Values.kind "postgres") }}
```

When the source secret has `DATABASE_URL`, the cronjob runs `pg_dump "$DATABASE_URL" | gzip | aws s3 cp …` to the cluster-wide `kuso-backup-s3` bucket using the same `<project>/<addon>/<ts>.sql.gz` shape.

### PlanetScale specifics

PlanetScale Postgres exposes a standard libpq DSN. No special driver needed; `pg_dump` works out of the box. Their docs confirm `pg_dump --no-owner --no-acl` works against their hosted instances (TODO: confirm before shipping).

The conn-secret synthesized from `spec.external.secretName` must carry `DATABASE_URL` — already the convention. No new keys.

### Files touched

- `operator/helm-charts/kusoaddon/templates/backup-cronjob.yaml` (+50 lines: external-postgres branch)
- `docs/EXTERNAL_DB_BACKUPS.md` (new, ~60 lines): "how to back up your PlanetScale (or Hetzner Cloud Postgres, Neon, Supabase) Postgres into kuso's S3"

### Smoke test

2. Set `spec.backup.schedule: "*/5 * * * *"`, retention 1 day.
3. Watch the CronJob fire, dump appear in S3, prune logic run after retention.

## Item 3 — Cron onFailure watcher

### Status check

Schema shipped v0.15.0. Watcher is the work.

### New package `server-go/internal/cronwatch/`

Mirrors `nodewatch.Watcher`. Key methods:
- `Run(ctx)` — main loop. Polls Jobs labeled `kuso.sislelabs.com/cron-name` every 30s.
- `dispatch(failure)` — POSTs HMAC-signed payload to the configured webhook + mirrors to notify.

### Payload

```json
{
  "project": "tickero",
  "service": "tickero-api",
  "cron": "refund-deadline-sweep",
  "jobName": "tickero-cron-refund-…-29661234",
  "exitCode": 1,
  "startedAt": "2026-05-27T03:00:00Z",
  "finishedAt": "2026-05-27T03:00:42Z",
  "logsURL": "https://kuso.tickero.bg/projects/tickero/services/tickero-api/crons/refund-deadline-sweep/runs/29661234"
}
```

### Signing

`X-Kuso-Signature: sha256=<hex>` over the raw body using the resolved secret value. Same convention notify webhooks already use — share the helper.

### Idempotency

In-memory `map[types.UID]bool` of dispatched Job UIDs. Failed Jobs prune via `failedJobsHistoryLimit`, so the map self-bounds.

### Boot wiring

`server-go/cmd/server/main.go`: start `cronwatch.NewWatcher(…).Run(ctx)` next to `nodewatch.Watcher` and `nodemetrics.Sampler`. Use the same lifecycle/shutdown pattern.

### Files touched

- `server-go/internal/cronwatch/{watcher.go,watcher_test.go,dispatch.go}` (~300 lines new)
- `server-go/cmd/server/main.go` (+8 lines: start the watcher)
- `server-go/internal/notify/` add `EventCronFailed` constant (+5 lines)

### Smoke test

1. Create a KusoCron that runs `sh -c 'exit 1'` every 2 minutes.
2. Configure `onFailure.webhookURL` to a requestcatcher.com URL.
3. Watch the webhook receive the signed payload on the next Job failure.

## Item 4 — Per-path `wakeOn` exclusion

### Schema

`KusoService.spec.sleep`:

```yaml
sleep:
  enabled: true
  afterMinutes: 30
  wakeOn:
    excludePaths: ["/api/v1/payments/notify", "/api/v1/webhooks/*"]
```

When sleep would otherwise idle the deployment, the operator keeps `replicas: 1` if any `excludePaths` are configured **on the service** — i.e. the whole deployment stays warm if it has ANY callback path that needs to be always-on.

### Why "whole deployment" not "per-path"

Kube doesn't route requests inside a single Deployment by path. The minimum viable shape is: if a path needs to be warm, the whole pod stays warm. To get per-path sleep, you'd need to split the service into two Deployments (callback-handler + everything-else). Out of scope.

### Mechanics

`internal/sleep/` (if it exists) or wherever scale-to-zero is implemented today — add the check: `if len(svc.Spec.Sleep.WakeOn.ExcludePaths) > 0, skip the scale-to-zero on this service`.

### Files touched

- `operator/config/crd/bases/application.kuso.sislelabs.com_kusoservices.yaml` (+15 lines: wakeOn block)
- `operator/config/crd/bases/application.kuso.sislelabs.com_kusoenvironments.yaml` (+15 lines)
- `server-go/internal/sleep/` (or equivalent) — bypass when excludePaths non-empty
- `docs/EDIT_SAFETY.md` — update the sleep row

### Smoke test

1. Configure a service with `sleep.enabled: true, afterMinutes: 1, wakeOn.excludePaths: ["/health"]`.
2. Wait 2 min idle. Confirm pod still running.
3. Remove the excludePaths. Wait 2 min idle. Confirm pod scales to 0.

## Tickero readiness post-v0.16.0

After v0.16.0 ships, `bileti-official` can migrate with:

```yaml
# kuso.yaml in bileti-official root
project: tickero
services:
  - name: api
    runtime: dockerfile
    dockerfile: Dockerfile.api
    port: 8080
    release:
      command: ["migrate", "-path", "/app/migrations", "-database", "${{ postgres.DATABASE_URL }}", "up"]
      timeoutSeconds: 600
    sleep:
      wakeOn:
        excludePaths: ["/api/v1/payments/notify"]
  - name: worker
    runtime: dockerfile
    dockerfile: Dockerfile.worker
    fromService: api  # reuse the api image
  - name: frontend
    runtime: dockerfile
    dockerfile: Dockerfile.frontend
    port: 3000
  - name: backoffice
    runtime: dockerfile
    dockerfile: Dockerfile.backoffice
    port: 3000
addons:
  - name: db
    kind: postgres
    external:
      secretName: tickero-planetscale  # user creates this manually with DATABASE_URL
    backup:
      schedule: "0 3 * * *"
      retentionDays: 14
  - name: cache
    kind: redis
  - name: queue
    kind: nats
    ha: true
  - name: storage
    kind: s3
    backup:
      schedule: "0 4 * * *"
      retentionDays: 14
crons:
  - name: refund-deadline-sweep
    service: api
    schedule: "0 * * * *"
    command: ["./bin/worker", "sweep-refunds"]
    onFailure:
      webhookURL: https://hooks.slack.com/services/…
      secretRef:
        name: tickero-slack-conn
        key: signing-secret
```

The only thing that requires manual setup outside the kuso config: creating the `tickero-planetscale` Secret with the PlanetScale DSN, and the `tickero-slack-conn` Secret with the webhook signing key. Both are one `kubectl create secret` away.
